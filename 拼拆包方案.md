## 1. 方案结论

SIO 负责 NoC CHI 与 Clink 之间的数据格式转换。

- TX 方向：NoC CHI 数据转换为 Clink 帧，称为拼包。
- RX 方向：Clink 帧还原为 NoC CHI 数据，称为拆包。

本方案支持 req、rsp、dat 这 3 类 NoC CHI 通道。

dat 允许分多拍传输到对端。拼包时，dat 不再要求一拍完整承载，而是作为 `dat_fragment` 填充当前 Clink 帧中未被 req/rsp 使用的 payload 资源，避免 800 bit 帧中的 segment 空闲浪费。

本方案采用如下 Clink 帧划分：

| 字段 | 数量 | 单个宽度 | 总宽度 | 用途 |
| --- | ---: | ---: | ---: | --- |
| `clink_reserved` | 1 | 45 bit | 45 bit | Clink 保留信息 |
| `header_payload` | 1 | 27 bit | 27 bit | 可承载 NoC 通道数据 |
| `segment` | 14 | 52 bit | 728 bit | 等宽数据分段 |
| 合计 | - | - | 800 bit | 固定 Clink 帧宽 |

关键参数如下：

```text
HEADER_WIDTH = 72 bit
HEADER_PAYLOAD_WIDTH = 27 bit
SEGMENT_NUM = 14
SEGMENT_WIDTH = 52 bit

72 + 14 * 52 = 800
```

本方案满足以下核心目标：

1. segment 等宽，且每个 segment 宽度大于 50 bit。
2. 同一帧必须支持同时传输 1 个 req 和 1 个 rsp。
3. dat 支持 668 bit 和 604 bit 两种包长，且允许跨多拍传输。
4. dat 按包顺序串接为连续 dat segment stream。若上一 dat 包只剩尾部 segment，且下一 dat 包已经可接收，则同一 Clink 帧可以连续承载“上一包尾部 segment + 下一包头部 segment”。
5. 若当前帧发送 req/rsp 后仍有空闲 segment，且存在 dat segment stream 待发送，则空闲 segment 优先承载 dat fragment。
6. dat 只能使用 segment，不使用 `header_payload`，以简化拼拆包逻辑。
7. `45 bit clink_reserved` 不承载 NoC 有效数据，也不包含通道类型、有效长度、segment 占用信息。
8. 通道映射固定，RX 侧通过帧内 `frame_type[4:0]` 选择解析方式和 dat 新包信息。
9. 除 45 bit Clink 保留信息外，所有未承载 NoC 有效数据的 bit 都补 0，并计入整帧补零。

## 3. 设计输入与约束

### 3.1 NoC CHI 通道宽度

本方案包含 3 个 NoC CHI 通道，各通道位宽如下：

| 通道 | 位宽 |
| --- | ---: |
| req | 158 bit |
| dat | 668 bit 或 604 bit |
| rsp | 70 bit |

### 3.2 Clink 帧约束

Clink 侧每次传输固定为 800 bit。

其中 45 bit 为 Clink 保留信息，必须放入 header，且不能承载 NoC 有效数据。

### 3.3 设计约束

Clink 帧划分和通道映射必须满足以下约束：

1. Clink 帧总宽度固定为 800 bit。
2. `clink_reserved` 固定为 45 bit，只承载 Clink 保留信息。
3. `header_payload` 可以承载 NoC 有效数据。
4. segment 用于承载 NoC 有效数据。
5. segment 必须等宽。
6. 每个 segment 位宽必须大于 50 bit。
7. NoC 通道数据可以拆分到多个 segment 中传输。
8. 同一 Clink 帧必须支持同时完整传输 1 个 req 包和 1 个 rsp 包。
9. dat 包允许拆成多个 dat fragment 分多拍传输。
10. dat 包之间按 segment 顺序拼接，允许一个 dat fragment 同时包含上一 dat 包尾部 segment 和下一 dat 包头部 segment。
11. 同一 Clink 帧允许同时承载 req、rsp 和 dat fragment。
12. 通道类型、有效长度、segment 占用信息不放入 45 bit `clink_reserved`。
13. 通道数据按低 bit 到高 bit 的顺序放入 payload 区域，低 bit 放入低地址。
14. dat 只能写入 segment，不能写入 `header_payload`。
15. 未使用的 `header_payload` 和未使用的 segment 必须补 0，并计入整帧补零。

## 4. Clink 帧资源划分

### 4.1 资源层级

800 bit Clink 帧分为两类资源：

```text
Clink frame[799:0]
    -> clink_reserved: 45 bit
    -> NoC payload: 755 bit
        -> header_payload: 27 bit
        -> segment[0:13]: 14 * 52 bit
```

其中：

- `clink_reserved` 是 Clink 保留资源，不参与 NoC 通道映射。
- `header_payload` 是 header 中可承载 NoC 数据的 payload 资源。
- `segment[0:13]` 是等宽 payload 资源。

### 4.2 地址顺序

Clink 帧低地址到高地址的字段顺序如下：

```text
clink_reserved -> header_payload -> segment[0] -> segment[1] -> ... -> segment[13]
```

800 bit Clink 帧整体划分如下图所示：

```text
低地址                                                               高地址
bit0                                                                  bit799
┌───────────────┬───────────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐
│clink_reserved │header_pay │ S0 │ S1 │ S2 │ S3 │ S4 │ S5 │ S6 │ S7 │ S8 │ S9 │S10 │S11 │S12 │S13 │
│45 bit         │27 bit     │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │ 52 │

```

对应 bit 范围如下：

| --- | --- |
| `clink_reserved` | `[44:0]` |
| `header_payload` | `[71:45]` |
| `segment[0]` | `[123:72]` |
| `segment[1]` | `[175:124]` |
| `segment[2]` | `[227:176]` |
| `segment[3]` | `[279:228]` |
| `segment[4]` | `[331:280]` |
| `segment[5]` | `[383:332]` |

| `segment[13]` | `[799:748]` |

### 4.3 写入顺序

通道数据写入 payload 区域时，按通道数据低 bit 到高 bit 的顺序依次写入低地址到高地址。拆包时按相同顺序还原。

dat 分多拍传输时，每个 dat fragment 按 dat 包内 segment 顺序连续传输；每个 segment 内仍按低 bit 到高 bit 放置。RX 侧按 fragment 顺序拼回完整 dat 包。

## 5. 通道映射规则

### 5.1 单通道承载能力

| 通道 | 有效宽度 | header 使用位置 | header 使用宽度 | segment 使用位置 | 通道分配宽度 | 通道局部补零 |
| --- | ---: | --- | ---: | --- | ---: | ---: |
| req | 158 bit | `header_payload[19:18]` | 2 bit | `segment[0:2]` | 158 bit | 0 bit |
| rsp | 70 bit | `header_payload[17:0]` | 18 bit | `segment[3]` | 70 bit | 0 bit |
| dat，单独发送 | 668 bit 或 604 bit | 不使用 `header_payload` | 0 bit | `segment[0:12]` 或 `segment[0:11]` | 676 bit 或 624 bit | 8 bit 或 20 bit |
| dat，与 rsp 同帧 | 668 bit 或 604 bit | 不使用 `header_payload` | 0 bit | `segment[0:2]`、`segment[4:13]` | 676 bit | 8 bit 或 72 bit |
| dat，与 req 同帧 | fragment | 不使用 `header_payload` | 0 bit | `segment[3:13]` | 最大 572 bit | 按 fragment 有效宽度计算 |
| dat，与 req/rsp 同帧 | fragment | 不使用 `header_payload` | 0 bit | `segment[4:13]` | 520 bit | 按 fragment 有效宽度计算 |

`header_payload` 不要求整块固定给某个通道。本方案中 `header_payload[26:22]` 固定承载 `frame_type`，`header_payload[19:18]` 承载 req 高 2 bit，`header_payload[17:0]` 承载 rsp 高 18 bit，`header_payload[21:20]` 可作其他用途；若没有其他用途，则补 0 并计入整帧补零。

为简化实现，dat fragment 只允许写入 segment，不允许使用 `header_payload`。因此 dat fragment 的尾部若不足一个 segment，剩余 bit 在该 segment 内补 0；`header_payload` 只由 rsp 或其他非 dat 用途使用，未使用时补 0。

局部补零只表示通道自身占用区域内未用完的 bit，不代表整帧补零。整帧补零见第 8 章。

req 固定分配 `segment[0:2] + header_payload[19:18]`。rsp 固定使用 `segment[3] + header_payload[17:0]`，不与 req 冲突。req 与 rsp 同时使用 header 中不同 bit 段，从而把 req/rsp 同帧占用 segment 数降到最小。

### 5.2 dat fragment 容量

dat 可以使用当前帧中未被 req/rsp 占用的 segment 资源。不同通道组合下，dat fragment 容量如下：

| 同帧 req | 同帧 rsp | dat fragment 映射 | dat fragment 最大宽度 |
| --- | --- | --- | ---: |
| 否 | 否 | `segment[0:13]` | 728 bit |
| 否 | 是 | `segment[0:2]`、`segment[4:13]` | 676 bit |
| 是 | 否 | `segment[3:13]` | 572 bit |
| 是 | 是 | `segment[4:13]` | 520 bit |

说明：

1. 若 dat 剩余未发送数据宽度大于当前 dat fragment 最大宽度，则本帧发送最大宽度，剩余部分在后续帧继续发送。
2. 若 dat 剩余未发送数据宽度小于当前 dat fragment 最大宽度，则本帧发送剩余部分，未使用 segment 或 header bit 补 0。
3. 当 dat fragment 映射区域不连续时，按低地址到高地址顺序依次写入。例如 rsp + dat 时，先使用 `segment[0:2]`，再使用 `segment[4:13]`。
4. dat fragment 按 segment 颗粒传输；若 dat 包最后一个 segment 未用满，则该 segment 内剩余 bit 补 0。

### 5.4 dat 包重组规则

dat 包有两种总长度：

- 668 bit。
- 604 bit。

当 dat 包被拆成多个 dat fragment 时，TX 必须按 dat 包内 segment 顺序发送，RX 必须按接收顺序拼接 dat fragment。668 bit dat 占用 13 个 segment，最后一个 segment 高 8 bit 补 0；604 bit dat 占用 12 个 segment，最后一个 segment 高 20 bit 补 0。

TX 侧可以把多个 dat 包按顺序串接成连续 dat segment stream。若上一 dat 包的剩余尾部 segment 数小于当前帧 dat fragment segment 容量，且下一 dat 包已经通过 valid/ready 握手被接收，则当前帧可以继续承载下一 dat 包的头部 segment。也就是说，一个 Clink 帧中的 dat fragment 可以跨越 dat 包边界，但跨越发生在 segment 边界上；上一 dat 包最后一个 segment 内的补零不会被下一 dat 包复用。

TX 侧 dat segment stream 缓冲深度按保证拼接且输出不断流的最小场景配置。最坏情况下，668 bit dat 占 13 个 segment，而 `req + rsp + dat` 同帧时可以发送 10 个 dat segment，因此上一 dat 包最多留下 3 个尾部 segment。若下一拍同时接收一个完整 668 bit dat 包，则需要缓存：

```text
3 个上一包尾部 segment + 13 个下一包 segment = 16 个 segment
```

因此 dat segment stream 缓冲最小深度取 16 个 segment。若小于 16，则无法在“上一包最大尾部 + 下一包完整输入”场景下同帧拼接；若大于 16，只增加抗突发余量，不是保证该功能所必需。

如果 RX 侧需要恢复原始 dat 包边界，`frame_type[1:0]` 必须在每个新 dat 包开始时提供 `dat_new_valid` 和该包长度。若同一帧可能承载“上一包尾部 segment + 下一包头部 segment”，则 `dat_new_valid` 表示下一包从当前包收齐后的剩余 segment 起始位置开始。

本方案默认同一方向上的 dat 包按序发送，只允许“前一个 dat 包尾部 segment 紧接后一个 dat 包头部 segment”的顺序拼接，不允许多个 dat 包乱序交织。若后续需要乱序或交织发送多个 dat 包，则帧外控制信息必须额外携带 dat 包标识，否则 RX 侧无法判断 fragment 属于哪个 dat 包。

例如，668 bit dat 在 req + rsp + dat 场景下可以拆成：

```text
第 1 拍：发送 dat[519:0]，使用 segment[4:13]
第 2 拍：发送 dat[667:520]，使用后续帧中的 dat fragment 区域
```

604 bit dat 在 req + rsp + dat 场景下可以拆成：

```text
第 1 拍：发送 dat[519:0]，使用 segment[4:13]
第 2 拍：发送 dat[603:520]，使用后续帧中的 dat fragment 区域
```

若第 2 拍 dat fragment 区域仍有剩余 segment，且下一 dat 包已经到达，则第 2 拍可以继续发送下一 dat 包的低地址 segment，填满可用 segment。

## 6. 调度策略

TX 侧每个 Clink 帧按如下优先级选择待发送通道：

1. 若 req 有待发送报文，必须选择 req。
2. 若 rsp 有待发送报文，必须选择 rsp。
3. 在 req/rsp 映射完成后，若当前帧仍有空闲 segment，且存在 dat 待发送数据，则用空闲 segment 承载 dat fragment。
4. 若没有 req/rsp 待发送，但存在 dat 待发送数据，则当前帧全部 segment 都用于承载 dat fragment，`header_payload` 不承载 dat。
5. dat 输入采用 valid/ready 握手。当内部 dat segment stream 缓冲空间不足以继续接收一个完整 dat 包时，TX 拉低 `dat_ready`，上游必须保持 `dat_vld` 和 `dat` 稳定，直到重新握手成功。

该策略保证：

1. 只要当前同时存在 req 和 rsp 待发送报文，二者一定能在同一个 Clink 帧内同时传输。
2. dat 不会阻塞 req/rsp。
3. dat 会尽量填充当前帧中未被 req/rsp 使用的 segment，减少 segment 空闲浪费。
4. dat 包边界不会强制终止当前帧；上一包尾部 segment 后面可以紧接下一包头部 segment。

## 7. 拼包与拆包流程

### 7.1 格式标记与帧外控制信息

45 bit `clink_reserved` 不包含通道类型、有效长度、segment 占用信息。因此，拆包模块不能通过 `clink_reserved` 判断当前帧承载了哪些 CHI 通道。

当前帧的通道组合类型和 dat 新包信息由帧内 `frame_type[4:0]` 标记，固定放在 `header_payload[26:22]`。该字段不占用 `clink_reserved`，也不占用 dat segment。

`frame_type[4:0]` 编码如下：

- `frame_type[4]`：req valid。
- `frame_type[3]`：rsp valid。
- `frame_type[2]`：dat fragment valid。
- `frame_type[1]`：dat new valid，表示当前 dat fragment 中有新 dat 包开始。
- `frame_type[0]`：dat size 604，仅当 `frame_type[1] = 1'b1` 时有效；`1'b1` 表示 604 bit，`1'b0` 表示 668 bit。

RX 侧根据 `frame_type[1]` 锁存新 dat 包长度，并按接收顺序累计 dat segment。604 bit dat 收满 12 个 segment 后输出，668 bit dat 收满 13 个 segment 后输出，因此不再依赖外部传入 `dat_fragment_valid_segment_num` 和 `dat_fragment_last`。

`frame_type[4:2] = {req_valid, rsp_valid, dat_fragment_valid}`，编码如下：

| `frame_type[4:2]` | 通道组合 | req valid | rsp valid | dat fragment valid | 对应映射 |
| --- | --- | ---: | ---: | ---: | --- |
| `3'b000` | 空帧 | 0 | 0 | 0 | payload 全部补 0 |
| `3'b001` | dat | 0 | 0 | 1 | `dat = segment[0:13]` |
| `3'b010` | rsp | 0 | 1 | 0 | `rsp = header_payload[17:0] + segment[3]` |
| `3'b011` | rsp + dat | 0 | 1 | 1 | `rsp = header_payload[17:0] + segment[3]`；`dat = segment[0:2] + segment[4:13]` |
| `3'b100` | req | 1 | 0 | 0 | `req = header_payload[19:18] + segment[0:2]` |
| `3'b101` | req + dat | 1 | 0 | 1 | `req = header_payload[19:18] + segment[0:2]`；`dat = segment[3:13]` |
| `3'b110` | req + rsp | 1 | 1 | 0 | `rsp = header_payload[17:0] + segment[3]`；`req = header_payload[19:18] + segment[0:2]` |
| `3'b111` | req + rsp + dat | 1 | 1 | 1 | `rsp = header_payload[17:0] + segment[3]`；`req = header_payload[19:18] + segment[0:2]`；`dat = segment[4:13]` |

RX 侧必须先根据 `frame_type[4:2]` 选择第 5 章中的固定映射，再从对应 payload 位置提取 req、rsp 和 dat fragment。由于映射由 `frame_type` 唯一确定，不需要额外传递 segment 占用字段。

当 `dat fragment valid = 1` 且当前 dat fragment 中有新 dat 包开始时，`frame_type[1]` 必须拉高并由 `frame_type[0]` 携带 `dat_size_604`。当 `dat fragment valid = 0` 时，这些 dat 相关字段无效并由 RX 忽略。

### 7.2 拼包流程

TX 方向拼包流程如下：

1. 从 NoC 侧选择待发送的 CHI 通道数据。
2. 按第 6 章调度策略确定当前 Clink 帧承载的通道组合，并生成 `frame_type[4:0]`。
3. 按第 5 章固定映射表确定 req/rsp 映射区域。
4. 若存在 dat 待发送数据，则将 dat fragment 写入当前帧剩余 segment 区域。
5. 将 45 bit Clink 保留信息写入 `clink_reserved`。
6. 将 NoC 通道数据按低 bit 到高 bit 的顺序写入低地址到高地址。
7. 对当前帧中未承载 NoC 有效数据的 payload bit 补 0。
8. 将 `frame_type[4:0]` 写入 `header_payload[26:22]`，生成固定 800 bit Clink 帧并发送到 Clink。

### 7.3 拆包流程

RX 方向拆包流程如下：

1. 从 Clink 接收固定 800 bit Clink 帧。
2. 从 `header_payload[26:22]` 提取 `frame_type[4:0]`。
3. 根据 `frame_type[4:2]` 和第 5 章固定映射表确定当前帧承载的 NoC 通道类型。
4. 按映射规则从 `header_payload` 和 segment 中提取 req/rsp，从 segment 中提取 dat fragment，低地址还原为通道低 bit。
5. 对 req/rsp，丢弃各自占用区域内的补零 bit 后上送 NoC。
6. 对 dat，将 dat fragment 按接收顺序拼接；根据当前 dat 包总长度收满 12 或 13 个 segment 后上送 NoC。
7. 解析 `clink_reserved` 中的 45 bit Clink 保留信息。

## 8. 补零计算

### 8.1 整帧补零

整帧可承载 NoC 数据的最大宽度为：

```text
NOC_PAYLOAD_WIDTH = 800 - 45 = 755 bit
```

整帧补零 bit 数量按以下公式计算：

```text
frame_padding = 755 - valid_noc_payload_width - used_metadata_width
```

其中：

- `valid_noc_payload_width` 是当前 Clink 帧实际承载的 req、rsp、dat fragment 有效数据总宽度。
- `used_metadata_width` 是 `header_payload` 中实际使用但不属于 NoC 数据的控制字段宽度。本方案中 `frame_type[4:0]` 固定占 5 bit。
- `frame_padding` 包括通道占用区域内未用完的 bit。
- `frame_padding` 也包括未使用的 `header_payload` 和未使用的 segment。
- 不允许 `valid_noc_payload_width + used_metadata_width > 755`。

### 8.2 示例

同帧 req + rsp 且没有 dat fragment 时，整帧补零为：

```text
755 - (158 + 70) - 5 = 522 bit
```

同帧 req + rsp + dat fragment 时，dat fragment 最大为 520 bit，整帧补零为：

```text
755 - (158 + 70 + 520) - 5 = 2 bit
```

同帧 req + dat fragment 时，dat fragment 最大为 572 bit，整帧补零为：

```text
755 - (158 + 572) - 5 = 20 bit
```

同帧 rsp + 668 bit dat 时，整帧补零为：

```text
755 - (70 + 668) - 5 = 12 bit
```

同帧 rsp + 604 bit dat 时，整帧补零为：

```text
755 - (70 + 604) - 5 = 76 bit
```

单独发送 668 bit dat 时，整帧补零为：

```text
755 - 668 - 5 = 82 bit
```

单独发送 604 bit dat 时，整帧补零为：

```text
755 - 604 - 5 = 146 bit
```

## 9. 设计说明

### 9.1 为什么需要 segment

Clink 帧宽为固定 800 bit，而 NoC CHI 各通道位宽不同。如果只把 800 bit 看成一整段 payload，TX/RX 需要额外传递每个通道的起始位置和长度，拆包逻辑也会变成可变解析。

定义等宽 segment 后，拼拆包逻辑可以退化为固定 segment 选择问题：

1. req/rsp/dat 占用固定数量的 segment，req/rsp 同时使用 `header_payload` 中不重叠的 bit 段以减少 segment 补零。
2. RX 侧只需要根据 `header_payload[26:22]` 中的 `frame_type[4:0]` 选择固定映射表并获取 dat 新包信息。
3. req、rsp、dat 的复用关系清晰，便于仲裁和时序实现。
4. 通道占用区域内的局部补零和整帧补零可以分开计算，便于比较不同划分方案。

### 9.2 为什么需要保证 req、rsp 一定能传输

req 和 rsp 通常位于请求与响应控制路径上。如果拼包方案允许 dat 占满 payload，可能导致 req/rsp 在链路侧被阻塞，进而影响事务推进和系统时延。

因此本方案把 req/rsp 作为调度优先级最高的通道：

1. 当 req 与 rsp 同时存在时，req 固定使用 `segment[0:2] + header_payload[19:18]`，rsp 固定使用 `segment[3] + header_payload[17:0]`，二者完成同帧传输。
2. req 单独出现或与 dat 同帧出现时，也固定使用 `segment[0:2] + header_payload[19:18]`。
3. dat 使用 req/rsp 固定位置之外的 segment，不阻塞 req/rsp。
4. req/rsp 的保底传输能力不依赖 dat 是否占用剩余 payload。

### 9.3 为什么选择 14 个 52 bit segment

新规格下，Clink 帧中可承载 NoC 数据的 payload 总宽度为 755 bit。

采用 `header_payload = 27 bit`、`segment = 14 * 52 bit` 后：

1. req 固定使用 `segment[0:2] + header_payload[19:18]`，不随 rsp 是否同帧而变化，便于 RX 使用固定位置拆包。
2. rsp 固定使用 `segment[3] + header_payload[17:0]`，分配宽度为 70 bit，可完整承载 70 bit rsp，且不与 req 冲突；`header_payload[26:22]` 固定承载 `frame_type`，`header_payload[21:20]` 可作其他用途或补 0。
3. req 与 rsp 合计有效数据为 228 bit。扣除 5 bit `frame_type` 后，即使把剩余 22 bit header 全部用于 req/rsp，也至少需要 `ceil((228 - 22) / 52) = 4` 个 segment。本方案 req/rsp 同帧正好占用 4 个 segment，达到 segment 占用下界。
4. req 与 dat 同帧时，dat 可使用 `segment[3:13]`，最大承载 572 bit，避免 req 固定位置之外的 segment 空闲浪费。
5. dat 与 rsp 同帧时，dat 可使用 `segment[0:2]` 和 `segment[4:13]`，最大承载 676 bit；668 bit dat 可一拍传完。
6. dat 与 req/rsp 同帧时，dat 可使用 `segment[4:13]`，最大承载 520 bit，避免 req/rsp 固定位置之外的 segment 空闲浪费。
