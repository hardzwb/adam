# 拼拆包验证方案

## 1. 验证目标

验证 `sio_clink_pack` 与 `sio_clink_unpack` 在同一套 Clink frame 映射下可以完成 NoC CHI `req`、`rsp`、`dat` 的端到端还原。

重点检查：

1. `req` 固定映射 `segment[0:2] + header_payload[19:18]` 后可被拆回 158 bit。
2. `rsp` 固定映射 `segment[3] + header_payload[17:0]` 后可被拆回 70 bit。
3. `frame_type[4:0]` 固定放在 `header_payload[26:22]`，编码为 `{req_valid, rsp_valid, dat_valid, dat_new_valid, dat_size_604}`。
4. `dat` 只使用 segment，不使用 `header_payload`。
5. 604 bit dat 收满 12 个 segment 后输出，668 bit dat 收满 13 个 segment 后输出。
6. dat 跨帧传输和“上一包尾部 segment + 下一包头部 segment”同帧拼接时，RX 能按 `frame_type[1:0]` 恢复包边界。

## 2. Loopback 顶层

RTL 顶层文件：

```text
rtl/sio_clink_loopback_top.v
```

连接关系：

```text
NoC input
  -> sio_clink_pack
      -> clink_frame_w / clink_frame_vld_o
  -> sio_clink_unpack
      -> NoC output
```

顶层假设 Clink 中间链路无反压。原因是当前 `sio_clink_pack` 没有 Clink-side ready 输入，无法根据 `sio_clink_unpack.clink_frame_ready_o` 反压停止出帧。因此 loopback 验证中，输出侧应保持：

```text
req_ready_i = 1
rsp_ready_i = 1
dat_ready_i = 1
```

如果后续需要验证 Clink 侧反压，需要给 pack 增加 Clink frame ready 输入，或在 loopback 顶层加入 frame skid buffer。

## 5. 覆盖场景

### 5.1 req/rsp only

输入：

```text
req_vld = 1
rsp_vld = 1
dat_vld = 0
```

期望：

```text
frame_type = 5'b11000
unpack req_o == pack req_i
unpack rsp_o == pack rsp_i
无 dat 输出
```

### 5.2 668 bit dat only

输入一个 668 bit dat 包。

期望：

```text
frame_type = 5'b00110
dat_new_valid = 1
dat_size_604 = 0
unpack dat_o == pack dat_i
dat_size_604_o = 0
```

说明：dat-only 帧有 14 个 segment 容量，668 bit dat 占 13 个 segment，因此可一帧完成。

### 5.3 604 bit dat only

输入一个 604 bit dat 包，`dat_i[667:604] = 0`。

期望：

```text
frame_type = 5'b00111
dat_new_valid = 1
dat_size_604 = 1
unpack dat_o[603:0] == pack dat_i[603:0]
dat_size_604_o = 1
```

### 5.4 req + rsp + 668 bit dat 跨帧

第一帧同时输入 req、rsp、668 bit dat。

期望：

```text
第 1 帧 frame_type = 5'b11110
第 1 帧 dat 容量 = 10 segment
第 2 帧 frame_type = 5'b00100
第 2 帧继续发送剩余 3 segment
最终 unpack dat_o == pack dat_i
```

### 5.5 old dat tail + next dat head 同帧

第一帧发送 `req + rsp + dat0`，只承载 dat0 前 10 个 segment。第二帧输入 dat1，同时承载 dat0 剩余 3 个 segment 和 dat1 头部 11 个 segment。

期望：

```text
第 1 帧 frame_type = 5'b11110
第 2 帧 frame_type = 5'b00110
第 3 帧 frame_type = 5'b00100
unpack 先输出 dat0，再输出 dat1
```

该场景检查 `frame_type[1:0]` 是否能在同一帧中描述新 dat 包起点。

## 6. 结果判定

功能模型对以下内容做严格比较：

1. 输出 req 序列等于输入 req 序列。
2. 输出 rsp 序列等于输入 rsp 序列。
3. 输出 dat 序列等于输入 dat 序列。
4. 输出 dat size 标记等于输入 dat 包大小。
5. 每个场景的 `frame_type[4:0]` 符合预期。

任一比较失败时，脚本会触发 `assert` 并打印失败场景名称。

## 7. 后续 RTL 仿真建议

如果接入 Verilator、VCS、Xcelium、Questa 等 RTL 仿真器，建议增加真正的 RTL testbench：

1. 实例化 `sio_clink_loopback_top`。
2. 产生 clock/reset。
3. 将 `req_ready_i/rsp_ready_i/dat_ready_i` 固定为 1。
4. 按第 5 章场景驱动 `req_i/rsp_i/dat_i` 和 valid。
5. 使用 scoreboard 保存输入事务，并在输出 valid 时逐项比较。
6. 增加断言检查：
   - `frame_type_o[4] == req_vld_o`
   - `frame_type_o[3] == rsp_vld_o`
   - dat 输出顺序不乱序
   - 604 bit dat 输出时 `dat_size_604_o == 1`
   - 668 bit dat 输出时 `dat_size_604_o == 0`

当前 Python 功能模型用于快速验证映射和包边界逻辑；RTL 仿真用于进一步验证时序、握手和寄存器行为。
