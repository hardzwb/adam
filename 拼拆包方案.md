# 拼拆包方案

## 1. 方案结论

SIO 负责 NoC CHI 与 Clink 之间的数据格式转换。

- TX 方向：NoC CHI 数据转换为 Clink 帧，称为拼包。
- RX 方向：Clink 帧还原为 NoC CHI 数据，称为拆包。

本方案采用如下 Clink 帧划分：

| 字段 | 数量 | 单个宽度 | 总宽度 | 用途 |
| --- | ---: | ---: | ---: | --- |
| `clink_reserved` | 1 | 45 bit | 45 bit | Clink 保留信息 |
| `header_payload` | 1 | 37 bit | 37 bit | 可承载 NoC 通道数据 |
| `segment` | 13 | 56 bit | 728 bit | 等宽数据分段 |
| 合计 | - | - | 810 bit | 固定 Clink 帧宽 |

关键参数如下：

```text
HEADER_WIDTH = 82 bit
HEADER_PAYLOAD_WIDTH = 37 bit
SEGMENT_NUM = 13
SEGMENT_WIDTH = 56 bit

82 + 13 * 56 = 810
```

本方案满足以下核心目标：

1. segment 等宽，且每个 segment 宽度大于 50 bit。
2. 同一帧必须支持同时传输 1 个 req 和 1 个 rsp。
3. `45 bit clink_reserved` 不承载 NoC 有效数据，也不包含通道类型、有效长度、segment 占用信息。
4. 通道映射固定，RX 侧通过帧外 valid 信息选择解析方式。
5. 除 45 bit Clink 保留信息外，所有未承载 NoC 有效数据的 bit 都补 0，并计入整帧补零。

## 3. 设计输入与约束

### 3.1 NoC CHI 通道宽度

NoC 侧包含 4 个 CHI 通道，各通道位宽如下：

| 通道 | 位宽 |
| --- | ---: |
| req | 162 bit |
| snp | 162 bit |
| dat | 606 bit |
| rsp | 70 bit |

### 3.2 Clink 帧约束

Clink 侧每次传输固定为 810 bit。

其中 45 bit 为 Clink 保留信息，必须放入 header，且不能承载 NoC 有效数据。

### 3.3 设计约束

Clink 帧划分和通道映射必须满足以下约束：

1. Clink 帧总宽度固定为 810 bit。
2. `clink_reserved` 固定为 45 bit，只承载 Clink 保留信息。
3. `header_payload` 可以承载 NoC 有效数据。
4. segment 用于承载 NoC 有效数据。
5. segment 必须等宽。
6. 每个 segment 位宽必须大于 50 bit。
7. NoC 通道数据可以拆分到多个 segment 中传输。
8. 同一 Clink 帧必须支持同时完整传输 1 个 req 包和 1 个 rsp 包。
9. 同一 Clink 帧允许同时承载不同 CHI 通道包。
10. 通道类型、有效长度、segment 占用信息不放入 45 bit `clink_reserved`。
11. 通道数据按低 bit 到高 bit 的顺序放入 payload 区域，低 bit 放入低地址。
12. 未使用的 `header_payload` 和未使用的 segment 必须补 0，并计入整帧补零。

## 4. Clink 帧资源划分

### 4.1 资源层级

810 bit Clink 帧分为两类资源：

```text
Clink frame[809:0]
    -> clink_reserved: 45 bit
    -> NoC payload: 765 bit
        -> header_payload: 37 bit
        -> segment[0:12]: 13 * 56 bit
```

其中：

- `clink_reserved` 是 Clink 保留资源，不参与 NoC 通道映射。
- `header_payload` 是 header 中可承载 NoC 数据的 payload 资源。
- `segment[0:12]` 是等宽 payload 资源。

### 4.2 地址顺序

Clink 帧低地址到高地址的字段顺序如下：

```text
clink_reserved -> header_payload -> segment[0] -> segment[1] -> ... -> segment[12]
```

对应 bit 范围如下：

| 字段 | bit 范围 |
| --- | --- |
| `clink_reserved` | `[44:0]` |
| `header_payload` | `[81:45]` |
| `segment[0]` | `[137:82]` |
| `segment[1]` | `[193:138]` |
| `segment[2]` | `[249:194]` |
| `segment[3]` | `[305:250]` |
| `segment[4]` | `[361:306]` |
| `segment[5]` | `[417:362]` |
| `segment[6]` | `[473:418]` |
| `segment[7]` | `[529:474]` |
| `segment[8]` | `[585:530]` |
| `segment[9]` | `[641:586]` |
| `segment[10]` | `[697:642]` |
| `segment[11]` | `[753:698]` |
| `segment[12]` | `[809:754]` |

### 4.3 写入顺序

通道数据写入 payload 区域时，按通道数据低 bit 到高 bit 的顺序依次写入低地址到高地址。拆包时按相同顺序还原。

## 5. 通道映射规则

### 5.1 单通道承载能力

| 通道 | 有效宽度 | 映射资源 | 分配宽度 | 局部补零 |
| --- | ---: | --- | ---: | ---: |
| req | 162 bit | 3 个 segment | 168 bit | 6 bit |
| snp | 162 bit | 3 个 segment | 168 bit | 6 bit |
| rsp | 70 bit | `header_payload + 1 个 segment` | 93 bit | 23 bit |
| dat | 606 bit | 11 个 segment | 616 bit | 10 bit |

局部补零只表示通道自身占用区域内未用完的 bit，不代表整帧补零。整帧补零见第 8 章。

### 5.2 固定映射表

由于通道类型、有效长度、segment 占用信息不放入 `clink_reserved`，payload 位置不能任意变化。TX 与 RX 必须使用同一张固定映射表，并由帧外 valid 信息选择对应行。

| 通道组合 | rsp 映射 | req 映射 | snp 映射 | dat 映射 | 整帧补零 |
| --- | --- | --- | --- | --- | ---: |
| rsp | `header_payload + segment[0]` | - | - | - | 695 bit |
| req | - | `segment[0:2]` | - | - | 603 bit |
| snp | - | - | `segment[0:2]` | - | 603 bit |
| dat | - | - | - | `segment[0:10]` | 159 bit |
| req + rsp | `header_payload + segment[0]` | `segment[1:3]` | - | - | 533 bit |
| snp + rsp | `header_payload + segment[0]` | - | `segment[1:3]` | - | 533 bit |
| req + snp | - | `segment[0:2]` | `segment[3:5]` | - | 441 bit |
| req + rsp + snp | `header_payload + segment[0]` | `segment[1:3]` | `segment[4:6]` | - | 371 bit |
| dat + rsp | `header_payload + segment[0]` | - | - | `segment[1:11]` | 89 bit |

### 5.3 不支持的同帧组合

以下组合无法在同一帧内完整承载：

| 通道组合 | 原因 |
| --- | --- |
| dat + req | 有效数据总宽度为 768 bit，超过单帧 NoC payload 上限 765 bit |
| dat + snp | 有效数据总宽度为 768 bit，超过单帧 NoC payload 上限 765 bit |
| dat + req + rsp | 有效数据总宽度为 838 bit，超过单帧 NoC payload 上限 765 bit |
| dat + snp + rsp | 有效数据总宽度为 838 bit，超过单帧 NoC payload 上限 765 bit |
| dat + req + snp | 有效数据总宽度为 930 bit，超过单帧 NoC payload 上限 765 bit |

## 6. 调度策略

TX 侧每个 Clink 帧按如下优先级选择待发送通道：

1. 若 req 有待发送报文，必须选择 req。
2. 若 rsp 有待发送报文，必须选择 rsp。
3. 在 req/rsp 已满足后，若剩余空间足够且 snp 有待发送报文，可以选择 snp。
4. dat 仅在没有 req 和 snp 待发送报文时选择。
5. dat 可以与 rsp 同帧发送。

该策略保证：只要当前同时存在 req 和 rsp 待发送报文，二者一定能在同一个 Clink 帧内同时传输。

## 7. 拼包与拆包流程

### 7.1 帧外控制信息

45 bit `clink_reserved` 不包含通道类型、有效长度、segment 占用信息。因此，拆包模块不能通过 `clink_reserved` 判断当前帧承载了哪些 CHI 通道。

TX 与 RX 之间必须存在帧外 valid 信息，至少包括：

- 当前帧是否包含 req。
- 当前帧是否包含 rsp。
- 当前帧是否包含 snp。
- 当前帧是否包含 dat。

由于本方案采用固定映射表，RX 侧拿到这些 valid 信息后即可确定每个通道对应的 payload 区域，不需要额外的 segment 占用字段。

### 7.2 拼包流程

TX 方向拼包流程如下：

1. 从 NoC 侧选择待发送的 CHI 通道数据。
2. 按第 6 章调度策略确定当前 Clink 帧承载的通道组合。
3. 按第 5 章固定映射表选择对应的 `header_payload` 与 segment。
4. 将 45 bit Clink 保留信息写入 `clink_reserved`。
5. 将 NoC 通道数据按低 bit 到高 bit 的顺序写入低地址到高地址。
6. 对当前帧中未承载 NoC 有效数据的 payload bit 补 0。
7. 生成固定 810 bit Clink 帧并发送到 Clink，同时发送对应的帧外 valid 信息。

### 7.3 拆包流程

RX 方向拆包流程如下：

1. 从 Clink 接收固定 810 bit Clink 帧。
2. 接收与当前帧对应的帧外 valid 信息。
3. 根据 valid 信息和第 5 章固定映射表确定当前帧承载的 NoC 通道类型。
4. 按该通道的映射规则从 `header_payload` 和 segment 中提取有效数据，低地址还原为通道低 bit。
5. 丢弃补零 bit。
6. 解析 `clink_reserved` 中的 45 bit Clink 保留信息。
7. 将还原后的 NoC CHI 通道数据发送到 NoC。

## 8. 补零计算

### 8.1 整帧补零

整帧可承载 NoC 数据的最大宽度为：

```text
NOC_PAYLOAD_WIDTH = 810 - 45 = 765 bit
```

整帧补零 bit 数量按以下公式计算：

```text
frame_padding = 765 - sum(channel_width[i])
```

其中：

- `sum(channel_width[i])` 是当前 Clink 帧实际承载的所有 NoC 通道有效数据总宽度。
- `frame_padding` 包括通道占用区域内未用完的 bit。
- `frame_padding` 也包括未使用的 `header_payload` 和未使用的 segment。
- 不允许 `sum(channel_width[i]) > 765`，否则该 Clink 帧无法完整承载当前通道组合。

### 8.2 示例

同帧 req + rsp 的整帧补零为：

```text
765 - (162 + 70) = 533 bit
```

其中，req 与 rsp 占用区域内的局部补零为 29 bit，剩余 504 bit 来自未使用的 segment。

同帧 dat + rsp 的整帧补零为：

```text
765 - (606 + 70) = 89 bit
```

## 9. 设计说明

### 9.1 为什么需要 segment

Clink 帧宽为固定 810 bit，而 NoC CHI 各通道位宽不同。如果只把 810 bit 看成一整段 payload，TX/RX 需要额外传递每个通道的起始位置和长度，拆包逻辑也会变成可变解析。

定义等宽 segment 后，拼拆包逻辑可以退化为固定 segment 选择问题：

1. 每个通道占用固定数量的 segment 或 `header_payload + segment`。
2. RX 侧只需要根据帧外 valid 信息选择固定映射表。
3. req、rsp、snp、dat 的复用关系清晰，便于仲裁和时序实现。
4. 通道占用区域内的局部补零和整帧补零可以分开计算，便于比较不同划分方案。

### 9.2 为什么需要保证 req、rsp 一定能传输

req 和 rsp 通常位于请求与响应控制路径上。如果拼包方案允许 dat 或 snp 占满 payload，可能导致 req/rsp 在链路侧被阻塞，进而影响事务推进和系统时延。

因此本方案把 req/rsp 作为调度优先级最高的通道：

1. 当 req 与 rsp 同时存在时，rsp 固定使用 `header_payload + segment[0]`，req 固定使用 `segment[1:3]`，二者完成同帧传输。
2. dat 只能在没有 req 和 snp 时发送，可以与 rsp 同帧发送。
3. req/rsp 的保底传输能力不依赖 dat 是否占用过相关 segment。
